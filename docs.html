API reference

Clearance is a tool call.

Send a product profile, get back a determination — auto-cleared, or routed with exactly what's missing — every component cited to its controlling clause. Available over HTTP and as an MCP tool.

Preview API. Schemas may change before general availability; we'll version and announce breaking changes.

Authentication

All requests send an API key over TLS. Keys are tenant-scoped and least-privilege (clear / construct / ratify / admin), hashed at rest, and support rotation and expiry. An admin key mints and rotates the rest.

X-API-Key: cw_live_…
Content-Type: application/json

Clear a product

POST /v1/clear — decompose a product, map each component to its controlling obligation and an approved control, and return a determination.

POST https://api.clearwayagents.ai/v1/clear
{
  "product": "Cross-border BNPL · US+MX",
  "jurisdictions": ["US", "MX"],
  "components": [
    { "ref": "Identity / CIP", "attributes": { "idv": "doc+selfie" } },
    { "ref": "AI underwriting", "attributes": { "model": "in-house" } },
    { "ref": "MX remittance", "attributes": { "corridor": "US-MX" } }
  ]
}
→ 200 OK
{
  "id": "clr_3a9f…",
  "decision": "DELTA_REVIEW",
  "catalog_version": "2026-06-15",
  "cleared": [
    { "ref": "Identity / CIP", "control": "kyc-standard", "citation": "31 CFR 1022.220" }
  ],
  "routed": [
    { "ref": "AI underwriting", "to": "Model Risk",
      "reason": "ECOA §1002.4 + SR 11-7 — no approved control" },
    { "ref": "MX remittance", "to": "Legal",
      "reason": "Reg E §1005.30 + state MTL — corridor not cleared" }
  ]
}

Read the catalog

GET /v1/catalog — list approved controls and their cited obligations at the current version. GET /v1/catalog/versions returns the version history.

Fetch a past clearance

GET /v1/clearances/{id} — returns the full, reproducible determination: inputs, matched controls, citations, and the pinned catalog version. This is the examination artifact.

Examiner packet

GET /v1/clearances/{id}/packet — the determination as a review-ready ledger: each obligation with its citation, a source-law link, the satisfying control or routed owner, and its status — plus a recall attestation and an audit_chain_verified flag.

Verify the audit log

The event log is a sha-256 hash chain. GET /v1/events/verify recomputes it and reports the first break, if any; GET /v1/events/anchor returns the current head commitment (count + head hash) to publish externally, so even a full-database compromise can't rewrite history undetected.

GET /v1/events/verify
→ 200 { "ok": true, "entries": 128, "broken_at": null }

Delta webhooks

Subscribe to be notified when a clearance produces a delta or when a routed item is ratified. Each delivery is HMAC-signed (X-Clearway-Signature: sha256=…) and sent only to a re-validated public endpoint (SSRF / DNS-rebind guarded). GET /v1/webhooks/deliveries returns delivery telemetry.

POST {your_endpoint}
X-Clearway-Signature: sha256=…
{
  "event": "clearance.recorded",
  "clearance_id": "clr_3a9f…",
  "routed": [ { "ref": "AI underwriting", "to": "Model Risk" } ]
}

As an MCP tool

Clearway exposes clear() over the Model Context Protocol, so an agent in your stack can request a determination directly.

tool clear(product, jurisdictions, components)
  → { decision, cleared[], routed[], catalog_version }

Errors

400
malformed product profile — a component is missing ref.
401
missing or invalid API key.
403
key lacks the required scope for this route.
409
catalog is mid-ratification; retry with backoff.
413
request body exceeds the size limit.
422
jurisdiction not supported in your plan.
429
rate limited — see Retry-After.
